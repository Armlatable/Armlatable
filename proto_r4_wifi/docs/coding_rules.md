# コーディング規約 (Coding Rules)

本プロジェクトにおけるコード作成・修正のガイドライン。

## 1. コメント (Comments)

### 目的
コードの意図を明確にし、ドキュメント生成ツール (Doxygen等) との互換性を保つ。

### ルール
- **Doxygen スタイル** を採用する。
- 冗長な装飾（`// =====...` 等の区切り線）は避ける。
- ブロックの開始・終了タグ（`@{`, `@}`）は使用しない。
- 終了タグは `**/` を使用する。

### 良い例 (Good)
```cpp
/**
 * @brief モーターを制御する
 * @param pwm PWM値 (-255 ~ 255)
 **/
void setMotor(int pwm) {
    // ...
}
```

### 悪い例 (Bad)
```cpp
// ==========================================
// モーター制御関数
// ==========================================
void setMotor(int pwm) {
    // ...
}
```

## 2. アーキテクチャ (Architecture)

### 設計原則
- **ハードウェア抽象化**: マイコンや通信方式の違いは `hardware/` ディレクトリ内のインターフェースクラスで吸収し、上位層 (`main.py`, `Controller`) に影響を与えないようにする。
- **設定の分離**: WIFI SSID, ピン配置, パラメータ等はコードにハードコードせず、マクロ定数や設定ファイル (`config.yaml`) に分離する。

### PC ↔ ロボット間通信
- **ドングル方式**: PC のネットワーク環境に依存しないよう、PC 側に通信用マイコン（ESP32等）を接続し、専用の通信路（UDP/ESP-NOW等）を確立する構成を推奨する。

## 3. Python コード

### スタイル
- [PEP 8](https://peps.python.org/pep-0008/) に準拠する。
- 型ヒント (`Type Hints`) を積極的に活用する。
- Docstring は Google Style またはシンプルな形式で記述し、Doxygen スタイルと親和性を持たせる（`@brief` 等の使用も可）。
